====
1. Attacker User product access

The attacker account only has low-privilege access in Jira Software Cloud.
Specifically:
- The attacker is not a member of the confluence-users-duyvic group.
- The attacker only has basic logged-in user access without any special project permissions.
- The attacker is not added to the target project (BRDBRD).

So the attacker should not be allowed to create or edit issues in that project.

2. Project type / permissions

- Project type: Bug Tracking (Jira Software board)
- Permission scheme: Modified so that initially "Browse Projects" = Any logged-in user (used only for reproduction)
- All other permissions remain default Jira Software permissions.

3. Attacker user project access

The attacker user has no project access to the target project (BRDBRD) except the temporary “Browse Projects” permission during testing.
After switching permissions back so only confluence-users-duyvic can access the project:
The attacker still cannot view the project from the UI
but
The attacker CAN STILL send the saved GraphQL request and create/edit issues, which is the vulnerability.
